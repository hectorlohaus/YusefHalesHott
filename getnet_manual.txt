Introducción
El presente documento entrega una descripción y una guía para la integración de los 
comercios clientes de Getnet en la plataforma de ecommerce Web Checkout, 
solución de pagos fáciles y seguros para compras en comercio electrónico. 
Esta integración permite realizar una comunicación directa y segura con el servicio de 
procesamiento de pagos Web Checkout. A continuación se detallan los requerimientos 
que el comercio debe cumplir para realizar la integración y la mensajería que se debe 
manejar en el ﬂujo de la transacción de pago.
Para completar la integración, el comercio deberá proporcionar algunos datos 
adicionales a través del envío de un Formulario de Validación que se encuentra 
disponible en Getnet.cl.

Con Web Checkout tu sitio obtiene una URL a la cual será redireccionado el tarjeta habiente para realizar el proceso transaccional.

necesario que tu sitio se conecte usando la versión TLS 1.2. La comunicación se realiza a través del protocolo de seguridad

completamente por Getnet.

contamos con canales seguros a través de TLS 1.2 que permiten una comunicación cifrada mediante la autenticación 
del comercio con un login y secretkey.

Crear una transacción
de venta
• Se necesita la URL donde se van a enviar los datos mediante el uso del método POST (Endpoint).
• Se debe usar JSON para el envío de los parámetros de autenticación, información del pago y datos de control.
• El tiempo de expiración debe ser una fecha válida en formato ISO 8601. Indica el tiempo límite que tiene el usuario para  
 
realizar el pago y debe ser de al menos 5 minutos superior a la fecha actual. Importante: El campo “expiration” es el  
 
 
tiempo máximo que se da para la sesión, si un usuario no realiza la operación la sesión queda pendiente hasta expirar. El  
 
tiempo recomendando es de 15 minutos.
• La URL de retorno (returnUrl) es utilizada para redirigir al usuario una vez terminada la operación. Se recomienda enviar  
 
en la URL un dato con el cual el sitio pueda identiﬁcar el registro, por ejemplo: https://mysite.com/response/{reference}.
• Se debe conﬁgurar en tu servidor una URL de notiﬁcación usando los puertos 80 o 443 y en caso de que el pago se decline  
 
o apruebe, reciba una notiﬁcación mediante POST de forma asíncrona. Importante: Esta URL de notiﬁcación debe ser  
 
 
entregada a Getnet para su conﬁguración a través del Formulario de Validación de Web Checkout.
• Es responsabilidad de los comercios el consumo de los servicios web bajo el método POST.
• Cuando las transacciones son aprobadas o rechazadas por el autorizador, se presenta el comprobante de venta al tarjeta 
 
habiente, sin embargo, éste no redirecciona de manera automática al comercio. 
• El comercio, a través del método GetRequestInformation, debe consultar el estado de la transacción. Este método puede  
 
ser consumido más de una vez.

5. Cuando el pagador retorna a tu sitio, pueden presentarse tres casos:
 
a. Pago Aprobado: El pago se realizó exitosamente y ha recibido esta información mediante la notiﬁcación del   
 
 
punto 3. En este caso, se recomienda mostrar un mensaje al cliente informando que, por ejemplo, el pedido será  
 
 
despachado.
 
b. Pago Rechazado: El pago no se realizó exitosamente y ha recibido esta información mediante la notiﬁcación del  
 
 
punto 3. En este caso, se recomienda mostrar un mensaje al cliente informando que el pago fue rechazado.
 
c. Aún no se ha recibido información sobre el estado del pago: En este caso, es necesario que se invoque al método  
 
 
getRequestInformation pasando como parámetro el requestID, el cual proporcionará información sobre el   
 
 
estado del pago.
6. Tipos de transacciones soportadas
 
a. Venta: Esta transacción permite ventas con tarjetas de crédito, débito y prepago. Para el caso de tarjetas de   
 
 
crédito emitidas en Chile, las ventas pueden ser con cuotas en las modalidades de Cuotas Emisor (ﬁnanciadas  
 
 
por el Banco Emisor de la tarjeta) o Cuotas Comercio (ﬁnanciadas por el comercio). Para el caso de las tarjetas  
 
 
internacionales, solo se acepta ventas sin cuotas.
 
b. Anulación: Esta transacción permite realizar una anulación del monto total de una venta previamente aprobada.  
 
 
Esta debe ser ejecutada durante el mismo día (hasta las 23:59:59). La Anulación se envía en línea al Banco   
 
 
Emisor de la tarjeta para su autorización. Aplica para cualquier transacción efectuada con tarjeta crédito, débito  
 
 
o prepago que se desee anular. Desde canales de e-commerce, la anulación puede realizarse en cualquier   
 
 
momento, no hay reglas deﬁnidas para el tiempo válido para el envío de una anulación. En el Panel Administrativo  
 
 
esta funcionalidad se denomina “Reverso transacción” y se puede identiﬁcar con el campo Referencia.
 
c. Devolución: Esta transacción permite realizar una devolución total o parcial de una venta previamente aprobada  
 
 
el día anterior. La Devolución se envía en línea al Banco Emisor de la tarjeta para su autorización, pero dependerá del  
 
 
Banco Emisor el momento de cuando se realizará el abono al tarjetahabiente.
 
 
Aplica para tarjeta de crédito, débito o prepago. En el Panel Administrativo esta funcionalidad se denomina   
 
 
“Reembolsar Transacción”.
 
d. Reversa: La Reversa es una transacción generada automáticamente cuando se produce un problema en la   
 
 
comunicación y no se recibe una respuesta del tiempo límite (timeout). La Reversa se envía en línea al Banco  
 
 
Emisor de la tarjeta para su procesamiento.

Requerimientos para
consumir nuestro servicio
AUTENTICACIÓN
La autenticación al servicio debe ser enviada sobre el objeto auth, el cual debe contener los siguientes atributos:
     El login y el secretKey son suministrados por Getnet.
     Esta estructura debe de ser enviada en cada petición realizada al servicio
     El seed no puede tener diferencia de más de 5 minutos (Esta es la fecha de creación de la sesión más 5 minutos).
ERRORES
El servicio de WebCheckOut retorna los códigos de respuesta para la autenticación fallida en el atributo message del 
objeto status.
Parámetro 
Descripción
login 
Identiﬁcador del sitio.
tranKey 
Llave transaccional, que está formada por la siguiente operación: 
 
Base64(SHA-256(nonce + seed + secretkey)). El nonce dentro de la 
 
operación es el original, es decir, el que no está codiﬁcado en Base64.
nonce 
Valor aleatorio para cada solicitud codiﬁcado en Base64.
seed 
Fecha actual, la cual se genera en formato ISO 8601.
Código 
Causa
100 
UsernameToken no proporcionado (Encabezado de la autorización malformado).
101 
Identiﬁcador de sitio no existe (Login incorrecto o no se encuentra en el ambiente).
102 
El hash de TranKey no coincide (Trankey incorrecto o mal formado).
103 
Fecha de la semilla mayor de 5 minutos.
104 
Sitio inactivo.

Errores frecuentes
MENSAJE DE ERROR AUTENTICACIÓN MAL FORMADA
• Se presenta cuando el sistema no detecta que se esté enviando login, tranKey, seed o nonce en la estructura auth 
enviada, también puede presentarse si se envían estos datos, pero de manera incorrecta, es decir sin el parámetro 
content-type “application/json” de manera que el servidor interpreta la petición como texto en vez de un arreglo de 
datos.
• Tus servidores requieren TLSv1.2 o superior para recibir la solicitud, debido a la norma PCI. Por favor, revisa el cifrado 
y el protocolo utilizado para conectar al servidor. Si usas Java, ten presente que solo las versiones después de la 8 
tienen soporte completo.
• En el proceso de autenticación nosotros revisamos el campo Created, este campo debe estar en el tiempo GMT o el 
tiempo local usando el tiempo de zona. Si obtienes esta respuesta, se debe a que tu tiempo no es preciso con el 
tiempo real. Nosotros solo permitimos 5 minutos de diferencia entre los tiempos. Puedes usar NTP para mantener la 
precisión del reloj.
• Mantén en mente que BASE64 debería ser para el raw output de la SHA256 y de acuerdo con todos los lenguajes de 
programación este puede ser requerido para conﬁgurar esta opción, por ejemplo. En PHP base64_encode (sha256(... , 
true)) este parametro retornaria el raw output para el SHA256 algorithm.
MÉTODOS DE INTERFAZ DE WEBCHECKOUT
El webservice contiene diferentes métodos donde se ejecutan operaciones de tipo solicitud - respuesta, estas requieren 
para cada una, parámetros de ingreso que son usados por las estructuras de datos con las que se procesan información 
para posteriormente retornar una respuesta.

WEB CHECKOUT VÍA LIGHTBOX
La alternativa de utilizar Web Checkout vía lightbox en lugar de redirección permitirá que el formulario de pago de Getnet aparez-
ca a través de un popup sobre tu sitio cada vez que el usuario presione el botón pagar. Esta modalidad permite al usuario conti-
nuar el proceso de pago sin ser redireccionado fuera de tu sitio web.
En el caso que se llame al servicio a través de un dispositivo móvil, el lightbox se redireccionará a una nueva página.
Cómo integrar Web Checkout vía Lightbox
Para importar el lightbox en tu sitio, debes utilizar la etiqueta <script> de HTML de la siguiente manera:
              <script src="https://checkout.getnet.cl/lightbox.min.js"></script>

ceb356920837e29f3e9fd51cf "

Al recibir la notiﬁcación su sitio debe:
Identiﬁcar operación: Con el identiﬁcador de la sesión requestId y la referencia reference identiﬁcan la sesión 
notiﬁcada. Con esta funcionalidad puede validar el estado de la operación notiﬁcada y conﬁrmar el estado de las 
transacciones en su sistema.
Validar autenticidad de la notiﬁcación: Puedes validar que se trate de una respuesta de Getnet haciendo un SHA-1  con 
los datos requestId + status + date + secretKey.
Si este valor coincide con el valor proporcionado por el signature, puedes autenticar la respuesta y proceder a asentar en 
la base de datos.
Nota: En ninguna circunstancia reveles el secretKey en ninguna parte accesible a los clientes o usuarios de su aplicación.
Luego de obtener el estado de la sesión, puede continuar con la regla de negocio de su sitio. (Ejemplo: Si la operación es 
aprobada, envía un correo electrónico al usuario.)
Otra manera de validar la autenticidad de la notiﬁcación, es una vez recibida esta, consultar con el getRequestInformation el 
estado de la transacción, de esta manera de hace una validación doble y se veriﬁca que el estado enviado en la notiﬁcación sea 
el correcto.
